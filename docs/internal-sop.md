# 内部维护 SOP (Internal Maintenance)

## 改动流程
1. 阅读 `architecture.md`，确认改动是否在模块边界内。
2. 更新 `architecture.md`（如涉及接口/模型变更）。
3. 实现代码，遵守下方编码约束。
4. 运行单元测试，全部通过后方可提交。
5. 更新 `changelog.md`，记录版本与改动摘要。
6. 若涉及依赖变更，先更新 `dependencies.md`。

## 编码约束
- 所有运行时 API 严禁产生额外的 GC 内存分配。
- 公共方法必须包含参数校验与空值防御。
- 禁止在模块内引入单例模式；使用框架提供的生命周期管理。
- 禁止跨模块直接引用；模块间通信必须通过 Command / Event。
- Core 包禁止引入对上层模块或游戏业务逻辑的依赖。
- 数学库与序列化层必须保持引擎无关性（不引用 UnityEngine 命名空间）。

## 错误处理
- 使用异常而非返回码表达错误（框架内部错误用异常，业务错误用结果对象）。
- 异常必须携带上下文信息（操作名称、关键参数）。
- 禁止吞掉异常；至少记录日志后再决定是否抛出。

## 测试要求
- 每个公共 API 至少有一个正向用例和一个异常用例。
- 测试命名：`<方法名>_<场景>_<预期结果>`。
- 测试必须可独立运行，不依赖外部状态或执行顺序。

## 日志规范
- Debug 级别：内部状态变化、流程跟踪。
- Warning 级别：可恢复的异常情况。
- Error 级别：不可恢复的错误，需要人工介入。
